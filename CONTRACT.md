# Harness template contract (how the sandbox dispatcher uses each directory)

The sandbox dispatcher (`scripts/sandbox-agent-shell.sh` in the terminal repo) is
GENERIC. For a harness it:

1. `git clone --depth 1` this repo, then copies `<harness>/.` into `/workspace`
   (so files like `<harness>/AGENTS.md` become `/workspace/AGENTS.md`).
2. MOVES `bootstrap.sh` and `launch.sh` out of `/workspace` into `/opt/tribes`, then
   runs `bootstrap.sh` from there **ONCE** (first boot, cwd still `/workspace`). A
   marker file kept in `/opt/tribes` then skips re-running it on later
   relaunches/reconnects. The control scripts never linger in `/workspace`.
3. Loops: runs `launch.sh` from `/opt/tribes` (which `exec`s the harness), and drops
   to a held `bash` between runs. `launch.sh` runs on **EVERY** launch. Only the
   scripts' location changed тАФ `/workspace` is left holding just the harness's real
   config (`AGENTS.md`/`CLAUDE.md` + dot-config), never the control scripts or marker.

## PRINCIPLE: config is REAL FILES in the dir, not generated by scripts

The dir's contents are copied verbatim into `/workspace`, so **commit the actual
config files** (`.codex/config.toml`, `.hermes/config.yaml`, `.openclaw/exec-approvals.json`,
`.claude.json`, тАж) as real, version-controlled files тАФ do NOT `cat >`-generate them
in bootstrap.sh. The scripts only fill the handful of values that genuinely can't be
committed (see below). This keeps each harness's config visible and editable in git.

### The ONLY runtime values (everything else is static and committed)

Put these PLACEHOLDER tokens in the committed config files and let `sed` fill them:

- `__TRIBES_PROXY__`  тЖТ `${API_BASE_URL}/llm/proxy`  (the metered LLM proxy base)
- `__TRIBES_TOKEN__`  тЖТ `$TRIBES_API_KEY`            (per-sandbox secret bearer)
- `__TRIBES_MODEL__`  тЖТ `$TRIBES_LLM_MODEL`          (default model id)
- `__TRIBES_THEME__`  тЖТ `light`/`dark` from `$TRIBES_THEME` (per-launch)

The ONE thing that needs a runtime fetch (not a sed): the **live model catalog** for
harnesses that can't auto-discover a custom provider (openclaw/opencode/pi embed the
`/models` list). Fetch it in bootstrap.sh and write it into the committed file.

## Files each `<harness>/` directory MUST contain

> `bootstrap.sh`/`launch.sh` are still committed in the harness dir (and still copied
> into `/workspace` first), but the dispatcher relocates them to `/opt/tribes` at boot
> so they never linger in the user's workspace. What each script must DO is unchanged.

- The harness's **real config files** (committed, with the placeholders above).
- `bootstrap.sh`  тАФ install the harness binary; `sed` the placeholders in the seeded
  config files; fetch + embed the model catalog IF the harness needs it; substitute
  `__HOST__`тЖТ`$HOSTNAME` (falling back to `hostname`) in AGENTS.md. Runs ONCE, as root, cwd `/workspace`, POSIX `sh`.
- `launch.sh`     тАФ `export` any **env-based** creds (claude ANTHROPIC_*, codex
  OPENAI_API_KEY, grok GROK_*); re-`sed` `__TRIBES_THEME__` for harnesses whose theme
  is a file value (grok/hermes) so a toggle takes effect on relaunch; then
  `exec <harness> <flags>`. Runs EVERY launch.
- `AGENTS.md`     тАФ copy of the repo-root `AGENTS.md`. Add `CLAUDE.md` (same content)
  if the harness reads CLAUDE.md.

Guard the sed/proxy steps so an unset proxy env leaves the harness on the user's own
key: skip filling proxy placeholders when `TRIBES_LLM_MODEL`/`API_BASE_URL`/`TRIBES_API_KEY`
are empty.

## CRITICAL: env-vars vs files split

`bootstrap.sh` runs in its OWN process that exits тАФ so any `export FOO=bar` it does
is LOST before the harness launches. Therefore:

- **Config FILES** (`.codex/config.toml`, `.hermes/config.yaml`, `opencode.json`,
  `.pi/agent/models.json`, `.claude.json`, `cline auth ...` which writes a file,
  etc.) тЖТ write them in **bootstrap.sh** (they persist on disk).
- **Config ENV exports** (`ANTHROPIC_BASE_URL`, `ANTHROPIC_AUTH_TOKEN`,
  `OPENAI_API_KEY`, `GROK_MODELS_BASE_URL`, `GROK_CODE_XAI_API_KEY`, тАж) тЖТ do them in
  **launch.sh**, right before `exec`, because only that process's env reaches the
  harness.

## CRITICAL: the proxy TOKEN must be refreshed every launch (restore-safety)

`TRIBES_API_KEY` is the one runtime value that can **change across boots of the same
disk**. A pause archives the disk to R2; a restore boots that disk again but the
control plane **RE-MINTS** the per-sandbox key тАФ it REVOKES the old token and injects
a fresh `TRIBES_API_KEY` on the boot cmdline. The proxy rejects the revoked token
(401), so a box restored with its first-boot token baked into a config file comes
back with a dead LLM key.

Therefore a **file-based** harness (one whose token lives in an on-disk config, not an
env export) MUST re-apply the live token in **launch.sh** on every launch тАФ not only
in `bootstrap.sh` (which runs once). The proxy URL, model, and embedded catalog do
NOT change on restore, so they stay seeded in `bootstrap.sh`; only the token is
refreshed per launch. Idiomatically, mirror the per-launch theme re-`sed`:

```sh
# Re-point the on-disk apiKey at the live cmdline token (the proxy token is
# tribes_sb_...). No-op on a cold boot; skipped on BYO/unset.
CFG=/workspace/.<harness>/config.json
if [ -n "$TRIBES_API_KEY" ] && [ -f "$CFG" ]; then
  sed -i "s|tribes_sb_[0-9A-Za-z]*|$TRIBES_API_KEY|g" "$CFG"
fi
```

`cline` (whose token lives in a binary-managed provider file) instead re-runs its
`cline auth ...` command in `launch.sh` every boot. **env-based** harnesses
(`claude`/`codex`/`grok`) already export the live `TRIBES_API_KEY` each launch, so
they are restore-safe by construction. Covered by `test/launch-token-refresh.test.sh`.

## Env vars available at runtime (injected by the control plane)

- `TRIBES_LLM_MODEL` тАФ the default model id to preselect (e.g. `deepseek-v4-flash`)
- `API_BASE_URL`     тАФ control-plane base; the metered LLM proxy is `${API_BASE_URL}/llm/proxy`
- `TRIBES_API_KEY`   тАФ the per-sandbox static bearer the proxy verifies (the proxy token)
- `TRIBES_THEME`     тАФ `light` or `dark` (the browser's theme at create time)
- `HOSTNAME`         тАФ the VM's public subdomain host (for AGENTS.md `__HOST__`)

The proxy URL is `proxy="${API_BASE_URL}/llm/proxy"`, the token is `"$TRIBES_API_KEY"`.
Model discovery for the picker is `GET ${proxy}/models` with `Authorization: Bearer $TRIBES_API_KEY`.

## Rules

- POSIX `sh` only (the dispatcher runs `sh ./bootstrap.sh` / `sh /workspace/launch.sh`).
- Idempotent + tolerant: a missing optional tool or a failed catalog fetch must not
  abort the script (`|| true`, fallbacks). The VM is the security boundary, so every
  harness must be fully non-interactive (auto-approve / trust the workspace).
- If `TRIBES_LLM_MODEL`/`API_BASE_URL`/`TRIBES_API_KEY` are empty, skip the proxy
  config gracefully (the harness falls back to the user's own key) тАФ mirror the old
  `configure_llm_proxy` guard `[ -n "$TRIBES_LLM_MODEL" ] && [ -n "$API_BASE_URL" ] || return 0`.
